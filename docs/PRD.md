## 1. 제품 비전

AllBook은 고객이 주변 웰니스·뷰티 업체를 검색하고, 서비스를 선택하고, 온라인으로 예약할 수 있는 플랫폼입니다. 업체 운영자는 예약·직원·서비스를 효율적으로 관리하고, 플랫폼 관리자(Admin)는 전체 생태계를 감독·운영합니다.

**장기 목표**: 호주 웰니스·뷰티 예약의 표준 플랫폼이 되며, 업종 확장을 통해 단일 코드베이스로 다양한 비즈니스를 지원합니다.

---

## 2. 문제 정의

### 고객 (Customer)

- 전화·메시지로 예약해야 해 불편하다.
- 영업시간·가용 슬롯을 한눈에 보기 어렵다.
- 여러 업종(마사지, 네일 등)을 각각 다른 방식으로 예약해야 한다.

### 업체 운영자 (Shop Owner)

- 수기 예약 관리로 이중 예약·노쇼가 발생한다.
- 직원별 스케줄·서비스 메뉴 관리가 분산되어 있다.
- 온라인 유입 채널이 없다.

### 플랫폼 운영자 (Admin)

- 입점 업체·품질·분쟁을 중앙에서 관리할 도구가 필요하다.
- 업종별 정책·수수료·노출을 유연하게 조정해야 한다.

---

## 3. 목표 및 성공 지표

### 비즈니스 목표

| 목표 | 설명 |
|------|------|
| 예약 전환율 | 샵 상세 → 예약 완료 비율 |
| 업체 온보딩 | 월 신규 입점 업체 수 |
| 재예약율 | 기존 고객의 재방문 예약 비율 |
| 운영 효율 | Admin·Shop Owner의 CS·수기 처리 감소 |

### 기술 목표

- 가용성 99.9% 이상 (Vercel + Supabase 기준)
- 핵심 예약 플로우 LCP 2.5초 이내
- 신규 업종 추가 시 스키마·코드 변경 최소화

## 5. 사용자 역할 및 권한

| 역할 | 코드 | 설명 | 접근 범위 |
|------|------|------|-----------|
| 고객 | `customer` | 예약·리뷰·프로필 | 본인 데이터 |
| 업체 운영자 | `shop_owner` | 샵·서비스·직원·예약 관리 | 소유/위임된 샵 |
| 직원 | `staff` | 스케줄·담당 예약 조회 | 소속 샵 (제한적) |

> **Admin은 초기 개발 범위에 포함**합니다. 후속 페이즈가 아닌 1차 릴리스 설계에 반영합니다.

## 7. 업종 확장 전략

초기에는 `massage` 카테고리만 노출하되, 데이터 모델과 UI는 **업종 중립적**으로 설계합니다.

| 업종 코드 | 표시명 (고객 UI) | Phase |
|-----------|------------------|-------|
| `massage` | Massage | 1 |
| `beauty` | Beauty | 2 |
| `nail` | Nail | 2 |
| `spa` | Spa | 2 |

업종별 차이(예: 마사지 therapist vs 네일 technician)는 **메타데이터·라벨**로 처리하고, 핵심 엔티티(Shop, Service, Staff, Booking)는 공유합니다.

## 8. 비기능 요구사항

### 보안

- Supabase Row Level Security (RLS) 필수
- Admin API는 서버 사이드에서만 권한 검증
- 개인정보(PII) 최소 수집, 호주 Privacy Act 고려

### 성능

- 목록·상세 페이지 ISR/캐싱 전략 적용
- 예약 슬롯 조회는 DB 인덱스·쿼리 최적화

### 접근성

- WCAG 2.1 AA 수준 목표 (고객 UI)
- 키보드 네비게이션, 스크린 리더 라벨

### 국제화

- Phase 1: 고객 UI 영어 고정
- Phase 2+: i18n 인프라 도입 검토 (관리자 UI는 한국어/영어 선택 가능성)

### 관측

- 에러 트래킹 (Sentry 등)
- Vercel Analytics, 핵심 비즈니스 이벤트 로깅
